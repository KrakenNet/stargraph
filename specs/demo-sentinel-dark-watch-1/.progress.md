## Learnings

- Harbor's `MLNode` supports sklearn/xgboost/onnx ONLY. YOLO/PyTorch models must be exported to ONNX for MLNode, or use a custom `YOLOInferenceNode(NodeBase)` for better OBB post-processing control.
- YOLO OBB (oriented bounding box) is the right architecture for SAR vessel detection — vessels are elongated/rotated. Ultralytics YOLO11/YOLO26 has built-in OBB support + ONNX export.
- CVE-rem demo proves the full pattern: docker-compose + bootstrap.py + state.py + harbor.yaml + real_nodes.py + serve_*.py + nautilus.yaml. Follow exactly.
- Nautilus has 8 adapter types: postgres, pgvector, elasticsearch, rest, neo4j, servicenow, influxdb, s3. For SDW: rest (AIS/GFW/Marine Regions), s3 (Sentinel-1 tiles), postgres (detection results/AIS buffer).
- Nautilus adapters are request/response only — NOT streaming. AIS WebSocket needs a buffer daemon that writes to Postgres; Nautilus queries the buffer.
- xView3-SAR: ~1000 scenes, 243K labels, ~300GB total. Each scene is ~29400x24400 px GeoTIFF. Must tile into 640x640 patches for YOLO. Demo needs only 10-20 scenes.
- xView3 labels are CSV with: detect_lat, detect_lon, is_vessel, is_fishing, vessel_length_m, confidence (HIGH/MEDIUM). Point annotations, not bounding boxes — need to synthesize OBB from vessel_length_m.
- Sentinel-1 GRD on AWS: `s3://sentinel-s1-l1c/` (eu-central-1), no-sign-request. Free access.
- Harbor ModelRegistry (SQLite) supports register -> alias -> load-with-hash-verify. Enables champion/challenger model gating for nightly retrain.
- Port offsets: CVE-rem uses 5439/5440/6390/9000. SDW should use 5441/5442/6391/9001.
- Demo code runs as `uv run --no-project python -m demos.<module>` — no separate pyproject.toml per demo.
- Existing directory is `demos/sentinel-dark-watch` (hyphens) but Python module needs underscores. Recommend rename to `demos/sentinel_dark_watch/`.
- AISStream.io API key already in `.env`. WebSocket endpoint: `wss://stream.aisstream.io/v0/stream`. Subscription includes bounding boxes + MMSI filters.
- Marine Regions WFS at `https://geo.vliz.be/geoserver/MarineRegions/wfs` provides EEZ boundaries as GeoJSON. Can pre-load at bootstrap.
- Global Fishing Watch APIs available free for non-commercial: Vessel API, Events API, 4Wings API (SAR detections + fishing effort). Requires separate API key registration.
- Ultralytics YOLO ONNX export gives ~43% speed boost over PyTorch inference. Harbor's ONNX session cache is thread-safe (one session per model_id+version).

## Learnings

- Previous learnings...
- Requirements scoped to 17 FRs + 8 NFRs. Deliberate exclusions: agent eval loop/DSPy prompt tuning (v2), Neo4j vessel graph (v2), vector embeddings (v2), GFW API integration (pre-load instead), multi-user workflow.
- Active learning routing (Fathom rule: low-confidence → analyst queue, high-conf/low-risk → auto-accept) is the simplest self-improvement mechanism beyond nightly retrain. Included as FR-17.
- xView3 labels are point annotations, not bounding boxes. OBB synthesis from vessel_length_m is a non-trivial data engineering step — train_detector.py must handle this conversion.
- Offline mode is a hard requirement (investor demos can't depend on network). Every external service needs a mock fallback: AIS fixture, LLM shim, pre-staged tiles.
- Risk scoring is rule-based (not ML) — keeps it explainable for investors and avoids a second model training dependency.
- ReportingNode and GeoContextNode are NodeBase subclasses that call DSPy internally (not DSPyNode — DSPyNode can only project state fields, can't interleave PostGIS/formatting). Everything else is deterministic code.
- Streamlit HITL communicates with Harbor via REST/WebSocket API (same pattern as CVE-rem watcher), not direct Python imports. Keeps UI decoupled from graph engine.
- PostGIS required (not plain postgres:16) — AC-5.2 uses ST_Contains/ST_Distance for EEZ boundary checks. Image: `postgis/postgis:16-3.4`.
- AIS correlation spatial radius (500m) is meaningless with ±30min window unless position is predicted forward. Vessel at 15kn moves 14km in 30min. Predicted-position matching required.
- xView3 Strait of Hormuz scene count may be <10. Contingency: expand AOI to broader Persian Gulf / Arabian Sea.
- pgvector removed from v1 scope — consistent with "vector embedding search = v2" out-of-scope. Eliminates port 5442 and extra container.
- NMS tile-boundary dedup and land-mask filtering are non-trivial — promoted to explicit FRs (FR-18, FR-19) rather than implicit sub-bullets of AC-3.4/3.5.
- FR-17 (Fathom active learning) promoted to High — central to the self-improvement narrative for investors.
- LLM output ACs made testable: AC-5.4 requires EEZ name + port distance + AIS status; NFR-8 = "no raw JSON"; AC-10.4 = "axes labeled, data points match stored metrics".
- Real-time SAR ingest (STAC/Copernicus API) explicitly out of scope — README mentioned it but demo uses pre-staged tiles only.
- NMS implemented with pure-Python Sutherland-Hodgman polygon clipping for rotated IoU — no Shapely dependency required. Falls back cleanly on empty OBB corners.
- AIS correlation uses greedy closest-pair matching on predicted positions, sorted by distance. Haversine for geo-distance. All detections default to dark_vessel=True if AIS query fails (conservative).

### Design Phase
- Custom `YOLOInferenceNode(NodeBase)` required — MLNode's `predict()` API can't handle tiling, OBB decode, or geo-coord transform. ONNX session still loaded via Harbor's `get_onnx_session()` cache.
- AIS buffer daemon pattern is the only viable integration: Nautilus adapters are request/response, AISStream.io is WebSocket streaming. Daemon writes to Postgres; Nautilus queries the buffer.
- PostGIS required for spatial queries: `ST_Contains` (EEZ point-in-polygon), `ST_Distance` (port/coast distance), GIST indexes. Plain postgres can't do this.
- AIS predicted-position matching critical: vessel at 15kn moves 14km in 30min. Spatial radius on raw last-known position is useless with ±30min time window.
- Fathom rule syntax from CVE-rem: `?n <- (node-id (id <node>)) (state (<field> <value>))` + `then: [{ kind: goto, target: <next> }]`. HITL gates: `kind: interrupt` + `timeout: null` + branch passthrough for response routing.
- InterruptNode raises `_HitInterrupt` — does NOT return a patch dict. Loop catches signal, transitions to `awaiting-input`, emits `WaitingForInputEvent`.
- SubGraphNode reuses parent run's `run_id`, event bus, checkpointer. Distinguished only by `branch_id`. Retrain sub-graph needs separate `RetrainState` but shares execution context.
- `serve_*.py` pattern: argparse wrapper, imports `harbor.serve.api.create_app`, pins capabilities, starts uvicorn. Thin by design.
- Natural Earth coastlines (not GSHHG) — smaller download, shapefile format, sufficient resolution for land-mask. geopandas reads directly.
- Graph has 15 nodes in main pipeline, 5 in retrain sub-graph. Simpler than CVE-rem's ~70 nodes. No need for Phase 0/6 or triggered graphs in v1.

### Design Revision (review fixes)
- Harbor's `DSPyNode` only projects state fields to DSPy inputs/outputs — cannot interleave deterministic work (PostGIS, report formatting). Nodes needing both must subclass `NodeBase` and call DSPy modules internally.
- Unconditional rules and conditional rules on the same node completion create ambiguous routing. Rules exiting a node must be mutually exclusive (state-guarded) or reduced to a single unconditional rule.
- CVE-rem triggered graphs use `# Trigger: cron (...)` header comments + separate YAML files. Scheduling is external: APScheduler in serve_*.py or system cron calling `POST /v1/runs`.
- CVE-rem harbor.yaml has `governance:` with Bosun pack mounts (budgets, audit, safety_pii, retries, routing, kill_switches). New graphs should include at minimum `harbor.bosun.budgets` + `harbor.bosun.audit`.
- IRDocument model confirms: `governance: list[PackMount]` defaults to empty list. `stores: list[StoreRef]` also defaults empty. Both are optional but should be explicit in harbor.yaml for documentation.
- ultralytics package version 8.x ships YOLO11 architecture — confusing versioning. Pin `>=8.3` with inline comment explaining this.
